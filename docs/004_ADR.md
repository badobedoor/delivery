# Architecture Decision Records (ADR)

**Version:** 1.0
**Last Updated:** 2026-07-05

---

## Purpose

This document records the major architectural decisions made throughout the evolution of the Hala Delivery project.

It is NOT a changelog or release notes.

It explains WHY important decisions were made.

Future contributors should understand these decisions before attempting to redesign any part of the system.

---

## ADR Format

Every ADR follows this structure:

```
- Decision
- Context
- Alternatives Considered
- Why They Were Rejected
- Final Decision
- Long-Term Impact
```

---

## ADR-001 — Order Editing Uses an Editing Session

### Decision

Order editing is performed through a temporary `OrderEditSession` rather than directly modifying persisted orders.

### Context

Customer Service may need to remove unavailable meals, add meals, change quantities, change required variants, or add notes. These operations should remain reversible until confirmed.

### Alternatives Considered

1. **Edit database rows directly.** — Rejected. Partial edits could leave the order in an inconsistent state.
2. **Mutate UI state directly without a session.** — Rejected. No clear source of truth; hard to cancel; hard to compare before/after.

### Final Decision

Introduce `OrderEditSession`. The editing session owns all temporary changes. Persistence occurs only when Apply is executed.

### Long-Term Impact

Reversible editing, local calculations, easier testing, cleaner architecture.

---

## ADR-002 — Stable Identity Uses IDs

### Decision

Business identity is always based on immutable identifiers (`restaurantId`, `categoryId`, `menuItemId`, `extraId`).

### Context

Names may change. Translations may change. Display formatting may change. Identifiers should never change.

### Alternatives Considered

**Using names.** — Rejected. Names are presentation; IDs are identity.

### Long-Term Impact

Stable relationships, reliable persistence, safer integrations.

---

## ADR-003 — Shared Restaurant Experience

### Decision

Customer and Admin share the same restaurant browsing experience.

### Context

Originally Admin was going to implement a separate mini-menu. This would duplicate meal cards, categories, meal configuration, pricing, and extras.

### Alternatives Considered

**Separate Admin implementation.** — Rejected. Large maintenance cost; duplicated business logic and UI.

### Final Decision

Extract reusable restaurant components. Shared components receive data. Parents own behavior.

### Long-Term Impact

One implementation, multiple applications, lower maintenance.

---

## ADR-004 — Shared Components Use Composition

### Decision

Shared components never know which application uses them.

### Context

Boolean feature flags such as `isAdmin`, `isCustomer`, `editable`, `showCounter` were considered.

### Alternatives Considered

**Boolean feature flags.** — Rejected. Components become coupled to specific applications.

### Final Decision

Composition. Parents inject behavior through props. Examples: `footer`, `onClick`, `actions`.

### Long-Term Impact

Reusable components, minimal coupling, cleaner APIs.

---

## ADR-005 — Theme Uses CSS Variables

### Decision

Shared components are theme-aware through CSS custom properties (`var(--color-...)`).

### Context

Customer uses light theme. Admin uses dark theme.

### Alternatives Considered

- **Duplicate components.** — Rejected.
- **Boolean flags (`isDark`).** — Rejected.

### Final Decision

CSS variables. Theme is provided by the parent container via inline styles or `:root` defaults.

### Long-Term Impact

One UI implementation, multiple themes.

---

## ADR-006 — Atomic Order Saving

### Decision

Order editing persistence uses a single PostgreSQL function (`apply_order_edit`).

### Context

Saving an order requires: delete old items, insert new items, update totals.

### Alternatives Considered

**Three independent REST requests.** — Rejected. Partial failure could corrupt data.

### Final Decision

Single PostgreSQL function wrapping all three operations in one transaction.

### Long-Term Impact

Guaranteed consistency, atomic persistence, safer editing.

---

## ADR-007 — Rule of Three

### Decision

Shared abstractions are extracted only after demonstrating real reuse across multiple consumers.

### Context

During development several helpers appeared extractable (e.g., `calculateEditingTotals`).

### Alternatives Considered

**Immediate extraction.** — Rejected. Only one consumer existed at the time.

### Final Decision

Keep local until multiple consumers require it.

### Long-Term Impact

Avoid premature abstraction, keep architecture simple.

## ADR-008 — size_name as Database Column

### Decision

Required variant information is persisted as a `size_name` TEXT column in `order_items`.

### Context

The `extra_groups` system uses `type="size"` to denote variant groups. The selected variant name must be preserved at order time because variant names, prices, and availability may change over time.

### Alternatives Considered

- **Store variant ID only.** — Rejected. The variant's name and price may change, breaking historical accuracy.
- **Derive from extra_groups at display time.** — Rejected. The variant configuration can change after the order is placed.

### Final Decision

Persist the resolved variant name as `size_name` at the moment of order placement and preserve it through editing.

### Long-Term Impact

Historical accuracy, reliable communication to drivers and customers, consistent editing experience.

---

## ADR-009 — Data Integrity Preservation Through Editing

### Decision

`menu_item_id` and `size_name` must be preserved through the entire edit flow: query → modal display → editing session → persistence.

### Context

Initial implementation of order editing lost `menu_item_id` (became NULL) and `size_name` (disappeared) because the query and mapper did not fetch or forward these fields.

### Alternatives Considered

**Reconstruct from UI at save time.** — Rejected. The data already exists in the database; the failure was at the query layer.

### Final Decision

Trace the complete data flow and ensure every field is selected, mapped, stored in session state, and included in the persistence payload.

### Long-Term Impact

Reliable data integrity through all editing operations.

## Future ADRs

Every significant architectural decision should receive a new ADR.

Do not rely on memory. Document the decision while the reasoning is still fresh.
